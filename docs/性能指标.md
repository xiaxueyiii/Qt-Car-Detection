# 2. 性能指标

本节用于课程报告中的“性能指标”部分。测试对象为基于 Qt 6 Widgets + C++ 的汽车检测桌面应用 CarDet，项目路径为 `E:\Desktop\FriD\suanfa\CarDet`。为避免编造数据，本文只给出可真实执行的测试方法、测试脚本和结果记录模板；具体数值应以本机运行 `scripts/performance_benchmark.py` 和可选 C++ ONNX Runtime benchmark 后输出的 CSV/JSON 为准。

## 2.1 响应时间测试

### 2.1.1 UI 响应时间定义

本项目中的 UI 响应时间主要指用户在 Qt 图形界面中触发操作后，界面开始反馈或功能逻辑返回所经历的时间。根据功能类型，可分为以下几类：

| 响应时间类型 | 定义 | 对应模块 |
|---|---|---|
| 数据集加载延迟 | 点击“扫描并保存”后，完成图片、标签、类别统计和图像标注表生成的时间 | `DatasetPage::scanDataset`、`DatasetPage::refreshImageTable` |
| 标注操作延迟 | 用户绘制、移动、缩放、删除矩形框后，画布更新和标注表同步的时间 | `AnnotationCanvas`、`AnnotationPage::updateBoxTable` |
| 推理启动延迟 | 点击“开始识别/检测”后，到 Python 子进程启动并开始输出日志的时间 | `InferencePage::startInference`、`InferenceViewModel::runInference` |
| 训练启动延迟 | 点击“开始训练”后，到 Python 训练脚本输出第一行日志的时间 | `TrainingPage::startTraining`、`TrainingViewModel::startTraining` |
| 日志响应延迟 | Python 子进程输出一行 stdout 后，Qt 日志窗口显示该行的实时性 | `TrainingViewModel::handleChunk`、`InferenceViewModel::handleChunk` |

由于 Qt GUI 中的真实鼠标点击和绘制延迟需要 Qt Test 或自动化 UI 工具才能完整测量，本项目采用两类测试方式：

1. 使用 Python 脚本模拟底层逻辑耗时，例如数据集扫描、标注框数据结构操作、QProcess 风格子进程启动、stdout 读取延迟。
2. 使用人工观察和屏幕录制验证界面流畅性，例如标注拖动是否卡顿、训练日志是否持续刷新、推理期间 UI 是否可交互。


### 2.1.3 标注操作延迟

标注操作涉及鼠标事件、坐标转换、矩形框列表更新和画布重绘。项目中的真实逻辑位于：

- `AnnotationCanvas::mousePressEvent`
- `AnnotationCanvas::mouseMoveEvent`
- `AnnotationCanvas::mouseReleaseEvent`
- `AnnotationCanvas::pushHistory`
- `AnnotationCanvas::undo`
- `AnnotationCanvas::redo`
- `AnnotationPage::updateBoxTable`

自动化脚本中的指标为：

`annotation_box_operation_simulation`

该指标模拟矩形框创建、移动、缩放、删除和历史栈维护，不代表真实鼠标拖动时的完整 GUI 渲染延迟，但可以反映标注数据结构操作的性能。报告中可表述为：

> 标注操作延迟采用底层数据结构模拟测试和人工交互观察相结合的方式。脚本模拟矩形框创建、移动、缩放、删除及撤销栈维护，GUI 画布流畅性通过人工连续拖动标注框观察确认。

### 2.1.4 推理启动延迟

推理启动延迟主要由 Python 解释器启动、脚本导入依赖、模型加载等组成。脚本提供两个相关指标：

| 指标 | 含义 |
|---|---|
| `python_process_start_to_first_stdout` | 启动 Python 子进程到读取第一行 stdout 的时间，近似 QProcess 启动反馈延迟 |
| `python_ultralytics_pt_end_to_end` | 端到端推理时间，包括 Python 启动、模型加载、图像读取、推理、绘图和写文件 |

Qt 中对应逻辑位于 `InferenceViewModel::runInference`，其使用 `QProcess` 调用 `scripts/infer_image.py`，并通过 `readyReadStandardOutput` 实时读取日志。

## 2.3 并发能力分析

### 2.3.1 QProcess 并发训练/推理能力

CarDet 使用 `QProcess` 调用 Python 脚本，训练、部署、推理分别有独立的 ViewModel：

| 模块 | ViewModel | 并发策略 |
|---|---|---|
| 训练 | `TrainingViewModel` | 单个训练页内部通过 `isRunning()` 防止重复启动训练 |
| 部署/量化 | `DeployViewModel` | 单个部署任务运行时禁用导出和量化按钮 |
| 推理 | `InferenceViewModel` | 单个推理页内部通过 `isRunning()` 防止重复推理 |

从代码看，每个 ViewModel 内部只维护一个 `QProcess` 对象，因此同一页面不支持同时启动多个同类任务。不同页面理论上可分别启动各自子进程，但训练和推理同时运行会竞争 GPU、CPU、内存和磁盘资源，不建议课堂演示时并发执行。

### 2.3.2 UI 线程是否阻塞分析

| 操作 | 是否容易阻塞 UI | 原因 |
|---|---|---|
| Python 训练 | 一般不阻塞 | 训练在 QProcess 子进程中执行，Qt 主线程只读取日志 |
| Python 推理 | 一般不阻塞 | 推理在 QProcess 子进程中执行 |
| ONNX 导出/量化 | 一般不阻塞 | 导出和量化在 QProcess 子进程中执行 |
| 数据集扫描 | 可能短暂阻塞 | 当前扫描逻辑在 UI 线程执行，数据集很大时可能卡顿 |
| 标注拖动 | 通常流畅 | 标注操作主要为内存坐标计算和 QWidget 重绘 |
| C++ ONNX Runtime 推理 | 可能阻塞 | `InferencePage::startInference` 中直接调用 `QuantizedOnnxRunner::run`，该调用发生在 UI 线程 |

因此报告中可以写：

> 项目将训练、导出、量化和 Python 推理放入 QProcess 子进程执行，避免深度学习计算直接阻塞 Qt 主线程。但数据集扫描和 C++ ONNX Runtime 推理目前仍在主线程内执行，当数据集规模较大或 ONNX 推理耗时较长时，可能出现短暂界面卡顿。后续可将这两类任务迁移到 `QThread` 或 `QtConcurrent` 中。

### 2.3.3 多任务调度情况

本项目属于单机桌面应用，不是服务器系统，因此“并发能力”不适合按 Web 服务 QPS 评价。更合理的评价角度是：

1. 同一页面是否防止重复启动任务。
2. 长耗时任务是否脱离 UI 线程。
3. 多个耗时任务同时执行时是否有资源竞争提示。

当前项目已经实现前两点的一部分，但尚未实现全局任务调度器。例如用户理论上可以在训练时切换到推理页启动推理，这会导致 GPU 资源竞争。改进建议是在 `MainWindow` 或统一任务管理器中记录全局任务状态，在训练运行时提示不要同时推理或量化。

## 2.5 GUI 流畅性分析

### 2.5.1 Qt 事件循环是否阻塞

CarDet 的界面采用 Qt Widgets 单主线程事件循环。为了保持 GUI 响应，项目将训练、导出、量化、Python 推理放入外部 Python 子进程中运行。对应代码如下：

| 功能 | 异步方式 | 代码位置 |
|---|---|---|
| 训练 | `QProcess` | `TrainingViewModel::startTraining` |
| ONNX 导出/量化 | `QProcess` | `DeployViewModel::startProcess` |
| Python 推理 | `QProcess` | `InferenceViewModel::runInference` |
| 日志刷新 | Qt 信号槽 | `readyReadStandardOutput` 到 `logMessage` |

因此，训练和 Python 推理过程中，界面仍可切换标签页、查看日志和结果。但以下操作仍可能在主线程中产生短暂停顿：

- 数据集目录很大时扫描图片和标签。
- 标注页加载超大分辨率图片。
- C++ ONNX Runtime 推理直接在 `InferencePage::startInference` 中执行。
- 结果可视化页一次性刷新大量数据库记录。

### 2.5.2 大图加载卡顿分析

标注页和推理页使用 `QPixmap` 或 `QImage` 加载图片：

- `AnnotationCanvas::loadImage`
- `InferencePage::showImage`

当图片分辨率很高或磁盘读取较慢时，图片解码和缩放可能造成短暂卡顿。当前项目主要面向课程数据集，图片规模较小，通常可以满足演示需求。后续优化建议：

1. 图片加载放到后台线程。
2. 对超大图片生成预览图。
3. 标注画布只在必要区域重绘。
4. 切换图片时预加载下一张图。

### 2.5.3 训练日志响应延迟

训练日志实时性由两个因素决定：

1. Python 脚本是否以无缓冲方式输出。
2. Qt 是否及时读取 QProcess stdout。

本项目中 `TrainingViewModel::startTraining` 使用：

```cpp
args << "-u" << scriptPath
```

其中 `-u` 表示 Python 使用 unbuffered stdout，能够减少日志缓冲延迟。`TrainingViewModel` 通过 `readyReadStandardOutput` 读取日志，并逐行显示在 GUI。

benchmark 指标：

`stdout_realtime_latency`

该指标通过子进程连续输出带时间戳的日志，父进程读取后计算输出与读取之间的延迟，近似训练日志响应延迟。

### 2.5.4 优化建议

| 问题 | 优化建议 |
|---|---|
| 数据集扫描可能阻塞 UI | 将扫描逻辑移动到 `QThread`，扫描过程中显示进度 |
| C++ ONNX Runtime 推理可能阻塞 UI | 将 `QuantizedOnnxRunner::run` 放入后台线程 |
| 结果表刷新数据较多 | 增加分页查询和按需加载 |
| 标注大图加载慢 | 使用缩略图缓存和异步加载 |
| 训练/推理并发资源竞争 | 增加全局任务状态管理，限制训练时同时推理 |
| 性能数据未自动入库 | 增加 `performance_runs` 表保存 benchmark 结果 |

从结果看，Python 端到端推理耗时明显大于直接 ONNX Runtime 前向推理耗时，主要原因是端到端测试每次都会启动 Python 进程、加载模型、初始化依赖、读取图片、绘制检测框并写入结果文件；而 `Session.run` 测试只统计模型前向计算本身。因此课程报告中应将二者区分：端到端耗时反映用户点击按钮后的实际等待时间，直接推理耗时反映模型推理核心计算性能。
